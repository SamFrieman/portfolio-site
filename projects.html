CoreRecon: Passive Reconnaissance Platform

FastAPI · React · Python · Docker · Redis · Celery
LIVE ON GITHUB
Security analysts spend 20–40 minutes per target manually stitching together DNS records, WHOIS data, SSL cert chains, subdomain lists, and security headers from 6–8 different tools. During M&A due diligence, this doesn't scale to 40 targets. I built CoreRecon to collapse that workflow into a single interface that returns a risk-scored report in under 30 seconds.

CoreRecon is a full-stack web application integrating 8 passive intelligence modules: DNS analysis, SSL/TLS chain validation, WHOIS lookup, subdomain enumeration via crt.sh and HackerTarget, technology fingerprinting via server headers, and security header grading. All modules run in parallel using Celery + Redis — that's how I hit the sub-30-second target despite 8 concurrent API calls.

I chose passive-only collection deliberately. Active scanning (Nmap, Nikto) would be faster and more comprehensive, but it creates legal exposure when assessing targets without explicit authorization — exactly the scenario I'd experienced during M&A work at WIA. Every technique in CoreRecon works from public sources only, making it safe to use in pre-authorization reconnaissance. I also built XSS and SQL injection input sanitization into all user-facing inputs — I'd seen too many security tools skip this because "only security people use it."
Reduces per-target reconnaissance time from ~35 min to <30 sec. SQLite persistence layer tracks 500+ malicious indicators with historical scan correlation — repeat targets surface immediately rather than requiring re-analysis. Automated PDF reporting removes the manual step of formatting findings for stakeholder delivery.

Bakery Encoder/Decoder: Malware Analysis Utility

JavaScript · Cryptography · Client-Side Only
FULLY OFFLINE CAPABLE
During incident response, analysts frequently encounter encoded payloads — Base64 strings in registry keys, ROT13 in phishing URLs, hex-encoded PowerShell. The workflow is: paste into CyberChef, guess the encoding, try again. For obfuscated malware with chained encodings, this is slow and error-prone. Worse, pasting potentially malicious content into a cloud-hosted tool is a data handling risk some organizations prohibit.

Bakery supports 50+ bidirectional encoding/cipher methods across 5 categories: Base encodings (Base64/32/16/85/58/62), web encodings (URL/HTML/Unicode), numeric representations (Binary/Octal/Hex), substitution ciphers (ROT1–25, Caesar, Atbash, Morse, Bacon), and transposition methods. The auto-detection engine uses weighted quality scoring rather than simple pattern matching — it scores each candidate decoding on entropy, printable character ratio, and keyword presence to achieve 85%+ accuracy on real-world samples.

Fully client-side JavaScript was a deliberate architectural choice. Incident responders working on compromised systems, or in air-gapped environments, need tools that work offline. No data ever leaves the browser — which also means no logging, no API keys to rotate, no server to maintain. <100ms processing latency is a side effect of local execution, not an optimization target.
Threat pattern detection engine flags 40+ malicious command indicators: PowerShell execution policy bypasses, system enumeration commands, known obfuscation utilities. This converts a decoding tool into an analyst assist — not just "what is this string" but "is this string dangerous."

SOC Automation Platform: Incident Response Orchestration

Python · SOAR Concepts · Threat Intelligence APIs
IN ACTIVE DEVELOPMENT
Alert fatigue isn't just a staffing problem — it's an architecture problem. When analysts must manually enrich every alert (IP lookup → WHOIS → VirusTotal → ticketing system), they burn decision bandwidth on mechanical steps before they've even started analysis. I've seen this pattern at Windward Consulting: the repetitive overhead was measurable, and it was getting worse as alert volume grew.

The SOC Automation Platform orchestrates common incident response workflows: automated indicator enrichment across threat intelligence APIs, contextual alert correlation, and structured ticketing with pre-populated investigation notes. The architecture is modular — each enrichment step is a pluggable handler that can be enabled, disabled, or rate-limited independently.

Rather than building a monolithic SOAR replacement (which would take years and a team), I designed for the 80% case: the repetitive enrichment steps that any analyst could describe in 5 minutes but that consume 30+ minutes per shift. The goal is to give analysts a head-start, not replace their judgment. Human escalation paths are first-class citizens in the design — the system explicitly surfaces when it doesn't know something.
Target impact based on Windward data: 3–5 hrs/week recovered per analyst. Demo environment in progress — the platform is being developed against realistic synthetic alert data to validate the enrichment pipeline before production readiness claims.

C2 Beacon Detector: Browser-Based Threat Hunting

JavaScript · Machine Learning · ThreatFox API
LIVE ON GITHUB
C2 beacon traffic is designed to blend in. A Cobalt Strike beacon with 60-second check-ins and 10% jitter looks like background noise to most network monitoring tools. Detecting it requires statistical analysis of timing patterns across thousands of connections — which means either expensive commercial tooling or hours of manual PCAP analysis in Wireshark. Neither is accessible to a threat hunter working a lead at 11pm on a personal machine.

C2 Beacon Detector analyzes network traffic for behavioral C2 signatures: consistent timing intervals with predictable jitter, payload size consistency, and beaconing regularity. It cross-references suspicious connections against the ThreatFox API and a local signature database of known framework patterns including Cobalt Strike, Metasploit Meterpreter, and Sliver. All analysis runs client-side in the browser — no backend, no data exfiltration risk for the traffic samples being analyzed.

MITRE ATT&CK technique mapping (T1071, T1572, T1095) was built in from the start, not added as an afterthought. This was a deliberate choice to make findings immediately actionable in enterprise environments where SIEM rules and detection playbooks are organized by ATT&CK ID. A finding that says "suspicious beaconing interval" is ambiguous; a finding that says "T1071.001 — Application Layer Protocol: Web Protocols" maps directly to existing detection logic.
Output in HTML, PDF, and JSON formats covers the three primary consumer use cases: analyst reports (HTML), management briefings (PDF), and SIEM integration (JSON). The browser-based architecture means zero deployment friction — share a URL, open a browser, start hunting.
